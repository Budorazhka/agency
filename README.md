# maindash-agency

Дашборд для агентства недвижимости. Фронтенд-only, без бэкенда — все данные мок. Сделан как демо/прототип для показа клиентам и внутренней разработки.

---

## Что это

CRM-дашборд для агентств недвижимости. Показывает воронку лидов, аналитику, объекты, сотрудников, рассылки. Поддерживает несколько ролей с разными правами доступа к разделам и действиям.

Продукт существует в нескольких вариантах (editions): agency, developer, realtor, internal. Этот репозиторий — вариант `agency`. Остальные варианты — отдельные папки рядом. Тип аккаунта задаётся в моке при логине, влияет на доступные разделы и тему оформления.

Две темы:
- **felt** — тёмно-зелёная, роскошная. Используется на всех страницах `/dashboard/*`.
- **standard** — светлая, бежевая. Используется на публичных страницах (вход, регистрация).

---

## Роли

Иерархия сверху вниз: Собственник → Директор → РОП → Менеджер. Маркетолог — боковая роль, не входит в иерархию.

| Роль | Код | Что видит и может |
|---|---|---|
| Собственник | `owner` | Всё. Единственный, кто может блокировать аккаунты и управлять партнёрами/городами. |
| Директор | `director` | Почти всё, кроме блокировки аккаунтов и управления партнёрами. Управляет командой, экспортирует данные, видит аналитику сети. |
| РОП | `rop` | Видит всех лидов своего офиса, управляет лидами, меняет правила раздачи лидов, видит аналитику лидов, управляет объектами. |
| Маркетолог | `marketer` | Видит аналитику лидов и рекламных кампаний, все стадии воронки, может добавлять источники лидов. Лиды не ведёт. |
| Менеджер | `manager` | Ведёт своих лидов, управляет объектами недвижимости. Видит только своё. |

## Функционал по разделам

### Главный экран (`/dashboard`)

Двухколоночная сетка в felt-стиле (тёмно-зелёный фон, золотые акценты).

Левая колонка:
- 4 карточки статистики: Лиды (сегодня / за 7 дней), Объекты (добавлено сегодня / за 7 дней), Сделки (за месяц / всего закрыто), Комиссии (с начала месяца / средний чек). Маркетолог вместо Комиссий видит карточку Реклама (потрачено, средний CPL).
- Для менеджера карточка Объекты не показывается, статистика считается только по его лидам.
- Навигационная плитка — карточки-ссылки на все доступные разделы (состав плитки зависит от типа аккаунта, конфиг в `src/config/dashboard-blocks.ts`).

Правая колонка:
- Виджет задач: счётчики просроченных и активных задач, список 4 просроченных лидов с именем и стадией.
- Виджет последних лидов: 6 последних лидов с источником, именем, стадией и временем создания.

### Лиды (`/dashboard/leads`)

Страница разная в зависимости от роли:

**Менеджер** — видит только своих лидов. Четыре счётчика: всего / новые / с задачами / просроченные. Кнопка перехода в покер лидов.

**Маркетолог** — видит только вкладку «Кабинет маркетолога» с аналитикой рекламных кампаний.

**РОП и выше** — видит кнопку перехода в покер лидов + блок аналитики (если есть право `view_lead_analytics`).

#### Покер лидов (`/dashboard/leads/poker`)

Полноэкранная канбан-доска. Лиды показаны в виде карточек, разбитых по стадиям воронки. Открывается поверх всего интерфейса (fixed position).

Стадии воронки (слева направо): Новые → Контакт → Квалификация → Предложение → Сделка (успех) / Отказ (провал).

Функционал:
- Фильтрация по менеджеру (выпадающий список, доступно РОП+). Менеджер видит только своих.
- Drag-and-drop карточек между стадиями (через @dnd-kit).
- Клик по карточке открывает правую панель деталей лида.
- Создание нового лида через кнопку «+» (форма с Zod-валидацией).

#### Детали лида (боковая панель)

Открывается по клику на карточку лида в покере.

Содержит:
- Основные данные: имя, телефон, email, источник, бюджет, стадия.
- Флаги срочности и важности (две иконки-маркера).
- Задача: дата дедлайна, кто поставил (собственник / директор / РОП), признак просрочки.
- История событий: лента действий по лиду (комментарии, смены стадий, звонки, задачи).
- Форма добавления комментария.
- Форма регистрации покупателя (кнопка внизу, если лид дошёл до нужной стадии).

#### Аналитика лидов и кампаний (вкладка маркетолога)

KPI-карточки: всего лидов, конверсия, показы, клики, CPL.

Таблица рекламных кампаний: название, канал (Яндекс.Директ, ВКонтакте, myTarget и т.д.), бюджет, потрачено, лиды, CPL, статус (активная / приостановлена / завершена).

CRUD кампаний: создать, редактировать, удалить. Форма с Zod-валидацией: название, канал, бюджет, даты, UTM-метки.

Статус кампаний: play/pause переключается кнопкой. Копирование UTM-ссылки.

Распределение лидов по стадиям воронки (мини-диаграмма рядом с таблицей).

### Продукт (`/dashboard/product`)

Страница-хаб с двумя модулями:

**МЛМ-аналитика** — открывает дашборд аналитики сети партнёров (тот же компонент что и `SupremeOwnerDashboardPage`, описан ниже в разделе «Аналитика сети»). Внутри ProductPage рендерится в светлом контейнере поверх felt-фона.

**Объекты** — переход на `/dashboard/my-properties` (страница базы недвижимости).

### Объекты (`/dashboard/my-properties`)

База объектов недвижимости. Данные хранятся в `localStorage` (ключ `agency.product.properties`), начальный набор — мок из `mock-data.ts`.

Вкладки по типу: Вторичка / Первичка / Аренда / Архив.

Функционал:
- Поиск по тексту.
- Фильтры: район, тип, статус продажи, метраж, цена, этаж, состояние. Панель фильтров открывается/закрывается.
- Переключение вида: таблица / карточки.
- Сортировка по дате (убывание / возрастание).
- Фильтр по агенту (для РОП+).
- Переключатель «Мои объекты / Все объекты» (для менеджера по умолчанию «Мои»).
- Булк-операции: чекбоксы на объектах, панель массовых действий (появляется при выборе).
- Алерты: уведомления по объектам (просроченные показы, незакрытые задачи и т.д.), фильтрация по типу алерта.
- Создание объекта через мастер (PropertyWizardDialog) — многошаговая форма с полями: тип, адрес, площадь, цена, этаж, состояние, фото.
- Редактирование объекта — тот же мастер в режиме edit.
- Удаление (одиночное и bulk).

Права: маркетолог видит объекты в read-only. Менеджер в режиме «Все объекты» тоже read-only.

### Персонал (`/dashboard/personnel`)

Организационная схема агентства в стиле колоды карт.

Каждый сотрудник — карточка в форме игральной карты с инициалами, ролью и цветовым акцентом (золото = собственник, синий = директор, оранжевый = РОП, зелёный = менеджер).

Карточки соединены линиями иерархии (вертикальные и горизонтальные линии через CSS/inline styles). Дерево строится рекурсивно через компонент `OrgNode`.

Клик по карточке — выбирает сотрудника. Справа появляется боковая панель с деталями:
- Контакты: телефон, email.
- Дата найма.
- Кнопки действий: редактировать, удалить, заблокировать/разблокировать.

Блокировка доступна только собственнику. Заблокированный сотрудник не может войти (проверяется в `AuthContext.login`).

Добавление сотрудника — кнопка «+», открывает форму с полями: имя, email, логин, роль, типы источников лидов, телефон. Валидация через `managerSchema` (Zod).

### Обучение / LMS (`/dashboard/lms`)

База обучающих материалов для сотрудников агентства.

Вкладки по типу контента: Статьи / Скрипты / Презентации / PDF / Курсы.

Поиск по названию и описанию материала.

Каждый материал — карточка с: название, описание, тип, целевая роль (все / менеджер / РОП / директор), время чтения/прохождения, статус (доступен / заблокирован).

Заблокированные материалы отображаются с иконкой замка — кликнуть нельзя.

Клик по карточке открывает детальный просмотр (статья / скрипт разворачивается на странице, PDF и презентации — ссылка для открытия).

Режим администратора (директор и выше): появляются кнопки редактировать / удалить на каждой карточке. Кнопка «Добавить материал» — открывает диалог `LMSAdminDialog` с формой: название, описание, тип, целевая роль, URL контента.

### Настройки (`/dashboard/settings`)

Страница с боковым меню. Набор вкладок зависит от роли:

| Вкладка | Кто видит |
|---|---|
| Мой профиль | Все |
| Компания | Директор и выше |
| Уведомления | Все |
| Передача лидов | РОП и выше |
| Менеджеры | РОП и выше |
| Тариф и оплата | Только собственник |
| Безопасность | Все |

**Мой профиль** — имя, email, телефон, смена пароля. Форма с `userSettingsSchema`.

**Компания** — название компании, юридические данные, логотип.

**Уведомления** — переключатели типов уведомлений (новый лид, задача, сделка и т.д.).

**Передача лидов** — правила автораспределения лидов между менеджерами. Можно настроить: режим раздачи (равномерный / по загрузке / ручной), приоритеты, источники для каждого менеджера.

**Менеджеры** — список менеджеров с возможностью добавить, редактировать, удалить. Та же форма что в Персонале.

**Тариф и оплата** — информация о текущем тарифе, дата следующего платежа, история платежей (всё мок).

**Безопасность** — смена пароля, активные сессии, двухфакторная аутентификация (UI-заглушки).

### Аналитика сети / Дашборд собственника (`/dashboard/city/:cityId/partner`)

Это основной аналитический дашборд для собственника всей сети. Доступен через раздел Продукт → МЛМ-аналитика или напрямую через карточку города.

Переключатель периода: неделя / месяц / квартал / год.

Блоки:
- **Статические KPI** — общие цифры по сети: партнёры, офисы, менеджеры, сделки за период.
- **Динамические KPI** — карточки с трендом: новые лиды, сделки, выручка, конверсия.
- **График лидов** — линейный график по дням за выбранный период.
- **Воронка (FunnelKanban)** — канбан по стадиям с числами.
- **Таблица лидербord** — ранжированный список партнёров или менеджеров по KPI. Сортировка по любой колонке.
- **Активность** — календарная тепловая карта активности (как GitHub contributions).
- **График активности** — столбчатый/линейный график активности по дням.
- **Состав активности** — пайчарт по типам активностей.
- **Топ источников** — горизонтальные бары топ-источников лидов.
- **Распределение активности партнёров** — кто из партнёров сколько делает.

Переключатель кабинета (`CabinetSwitcher`) — позволяет смотреть данные как конкретный партнёр или офис.

Кнопка «Персональные инсайты» — открывает диалог с рекомендациями по конкретному партнёру.

Встроенная почта (`MailingsInbox`) — входящие рассылки прямо в дашборде.

Панель «перерыв» (`GamesBreakPanel`) — мини-игра для отдыха (пасьянс или похожее).

---

### Город (`/dashboard/city/:cityId`)

Страница управления конкретным городом сети. Доступна собственнику.

Показывает партнёров города: список, их роли, разрешения.

Управление разрешениями партнёра — можно менять набор разрешений и ролей через диалог. Изменения диспатчатся в `DashboardContext`.

Добавление нового партнёра в город.

Переключение на другие города той же страны через выпадающий список.

---

### Рассылки города (`/dashboard/city/:cityId/mailings`)

Страница создания и управления рассылками для конкретного города.

---

### Регистрация агентства (`/register`)

Многошаговая онбординг-форма для нового агентства. Поля: название компании, email, телефон, сайт, описание. Валидация через `agencyRegistrationSchema`.

---

### Страница входа (`/login` или `/`)

Форма логина: поле «Логин» и «Пароль». Валидация через `loginSchema`.

При успешном входе — редирект на `/dashboard`. При заблокированном аккаунте — сообщение об ошибке. При неверных данных — сообщение «Неверный логин или пароль».

Сессия сохраняется в `localStorage`, поэтому при перезагрузке страницы пользователь остаётся залогиненным.

# Для айтишника

## Запуск

```bash
npm install
npm run dev
```

Открывается на `http://localhost:5173`. Используется HashRouter, поэтому все URL выглядят как `/#/dashboard/leads`.

Сборка:
```bash
npm run build
```

Результат в папке `dist/`. Это статика, можно раздавать с любого веб-сервера или из S3.

Линтер:
```bash
npm run lint
```

    mock/           — leadFactory.ts: генерация тестовых лидов

## Провайдеры и их порядок

В `main.tsx` провайдеры вложены в такой последовательности (снаружи внутрь):

```
StrictMode
  RootErrorBoundary       — ловит всё что упало до рендера
    AuthProvider          — currentUser, login, logout, блокировки
      MockProvider        — сценарий мок-данных, simulateNetworkDelay
        LoadingProvider   — глобальный оверлей загрузки
          HashRouter
            ThemeProvider — тема (felt/standard) по текущему pathname
              DashboardProvider — данные дашборда
                LeadsProvider   — пул лидов
                  Routes        — маршруты
```

ThemeProvider должен быть внутри HashRouter, потому что использует `useLocation()`.

## Как добавить новый маршрут

1. Создать компонент страницы в нужной папке `src/components/...`
2. Импортировать его в `src/main.tsx`
3. Добавить `<Route>` внутри блока `<Route element={<RequireAuth />}>` (если страница требует авторизации) или внутри `<Route element={<PublicLayout />}>` (если публичная)
4. Добавить ссылку в сайдбар в `App.tsx` если нужно

Пример добавления защищённого маршрута `/dashboard/reports`:
```tsx
// main.tsx, внутри RequireAuth → /dashboard
<Route path="reports" element={<ReportsPage />} />
```

## Как добавить новую роль

1. В `src/types/auth.ts` добавить значение в `UserRole`
2. В `src/types/auth.ts` добавить массив разрешений в `ROLE_PERMISSIONS`
3. В `src/types/auth.ts` добавить описание причины отказа в `PERMISSION_DENIED_REASON` для каждого нового действия
4. В `src/lib/permissions.ts` добавить метку в `ROLE_LABEL` и цвет в `ROLE_COLOR`
5. В `src/context/AuthContext.tsx` добавить мок-пользователя в `MOCK_USERS` для тестирования

## Как добавить новое разрешение (PermissionAction)

1. В `src/types/auth.ts` добавить строку в тип `PermissionAction`
2. Прописать это действие в нужных ролях в `ROLE_PERMISSIONS`
3. Добавить текст причины отказа в `PERMISSION_DENIED_REASON`
4. Использовать в коде: `const { allowed, reason } = usePermissionProps('моё_действие', currentUser.role)`

## Темизация

Две темы определяются в `src/context/ThemeContext.tsx`:
- `felt` — включается когда pathname начинается с `/dashboard` или содержит `/partner`, или пользователь имеет тип `internal`
- `standard` — всё остальное

ThemeProvider не пишет ничего в DOM напрямую. Он только предоставляет контекст `{ theme, isFeltStyle }`. Класс `app-theme-felt` вешается на div-обёртку в `App.tsx`.

CSS-переменные felt-темы (`--felt-surface`, `--felt-border`, `--felt-text-primary` и т.д.) определены в `src/index.css` под селектором `.app-theme-felt`.

Чтобы компонент реагировал на тему — подключить хук `useTheme()` из `src/hooks/useTheme.ts`:
```tsx
const { isFeltStyle } = useTheme()
```

## Мок-данные и сценарии

Реального бэкенда нет. Все данные генерируются в браузере.

`MockProvider` (`src/providers/MockProvider.tsx`) хранит текущий сценарий и предоставляет:
- `scenario` — текущий сценарий (строка)
- `setScenario(scenario)` — сменить сценарий
- `simulateNetworkDelay(ms)` — промис с задержкой для имитации сети
- `simulateError(type)` — сгенерировать ошибку нужного типа

Сценарии (`MOCK_SCENARIOS`):

| Сценарий | Лидов | Новых | Просроченных | Конверсия |
|---|---|---|---|---|
| `default` | 120 | 15% | 10% | 15% |
| `many-new-leads` | 200 | 40% | 5% | 12% |
| `overdue-tasks` | 150 | 10% | 35% | 8% |
| `high-conversion` | 100 | 20% | 5% | 35% |
| `empty-pipeline` | 20 | 50% | 0% | 25% |
| `stress-test` | 1000 | 25% | 15% | 10% |

Лиды генерирует `src/lib/mock/leadFactory.ts`. Функция `generateLeads(count, options)` создаёт указанное количество лидов с рандомными, но реалистичными данными.

`LeadsContext` (`src/context/LeadsContext.tsx`) читает сценарий из MockProvider и на его основе генерирует пул лидов при инициализации.

Чтобы добавить новый сценарий — добавить запись в `MOCK_SCENARIOS` в MockProvider и обновить логику в LeadsContext чтобы он реагировал на новые параметры.

## Error Boundaries

Три уровня:

1. `RootErrorBoundary` в `main.tsx` — ловит всё что упало до рендера дерева. Показывает простой текст с сообщением об ошибке.
2. Модульные boundaries в `src/components/common/ModuleErrorBoundary.tsx` — по одному на каждый крупный раздел (Leads, Dashboard, Product, Personnel, Settings). В dev-режиме показывают стек трейс, в prod — заглушку с кнопкой перезагрузки.
3. `RuntimeErrorBoundary` — для точечной защиты конкретных компонентов внутри раздела.

В `import.meta.env.DEV` проверяется режим сборки. В dev показывается больше деталей.

## Валидация форм

Схемы в `src/lib/validation/schemas.ts` (Zod). Список схем:
- `loginSchema` — форма входа
- `agencyRegistrationSchema` — регистрация агентства
- `leadSchema` — создание/редактирование лида
- `managerSchema` — создание/редактирование менеджера
- `userSettingsSchema` — настройки профиля (включая смену пароля)
- `campaignSchema` — рекламная кампания

Хук `useFormValidation` (`src/hooks/useFormValidation.tsx`) — обёртка над react-hook-form + zodResolver. Сигнатура:
```tsx
useFormValidation(schema, { onSubmit?, onError? })
```

Первый аргумент — Zod-схема, второй — опции. Не объект с полем `schema` — именно сама схема первым аргументом.

Вспомогательные утилиты для форм в `src/lib/validation/utils.ts`.

## Авторизация

`AuthContext` (`src/context/AuthContext.tsx`) хранит `currentUser` в `localStorage` (ключ `agency.auth.current-user`). При перезагрузке страницы пользователь остаётся залогиненным.

Методы:
- `login(login, password)` — возвращает `'ok' | 'blocked' | 'invalid'`
- `enterAs(accountType, role)` — войти без пароля как любая роль (для демо)
- `logout()` — выйти
- `toggleBlockUser(userId)` / `isUserBlocked(userId)` — блокировка (только у role=owner)

Реального API нет. Логин проверяется по массиву `MOCK_USERS`.
